# 方案 v5：会话身份三侧统一到 (name, project) 复合键

日期：2026-07-19 PDT
版本：v5（v1 拦 3 条、v2 拦 2 条、v3 拦 3 条、v4 拦 1 条，均已吸收）
仓库：/Users/tommyclaw/AIAgent/plugins（agent-meeting）+ /Users/tommyclaw/AIAgent/AMBridge

## 0. 背景（已确证事实）

**project 是功能性身份，不是展示字段。** 2026-07-13 引入。起因：项目身份原从仓库目录名推导，而目录名是每台机器 clone 时的偶然选择——同一项目在不同机器上 clone 成不同文件夹名，手机端即裂成多个项目卡（实测 wda-v3 裂成三份、837 条消息分散）。doc12 确立显式 `--proj` 为权威身份，并否决「从 git remote URL 派生」（remote 是可变定位符而非身份）。

**现有分层是有意设计，不是错配。** 0.8.54（40160a8）把 1:1 房间寻址改为只按名字，因为当时 project 会漂移；0.8.55（90169a9）引入显式 `--proj` 后 project 才稳定，sessions 方敢用 (project, name) 复合主键。

**project 在 AMBridge 侧有 6 个功能性消费点**，无一为展示；delete 去重丢 project 会不可逆误删同名 agent，worker 归属已实际犯过错归 bug。

**已否掉**：「名字全局唯一 + project 降级展示字段」。doc12 从未评估过；会打破同名跨项目共存、控制指令路由、游标隔离（backlog 重放）、群成员身份（掉群）。

**两种失效模式（分开处理）**：模式 A（历史）脏候选下的正常投递——probeproj 当年是 roster 排序下唯一候选，投递逻辑没错、候选是脏的；模式 B（当前）同名并存下二义 dead-letter——CLI 多候选 SystemExit → amp 静默丢弃，PWA 仍显示 Delivered，消息丢失而非错投。

**现象二已定论**：那一帧从未发出，断在手机 PWA。推导：中继先持久化再广播、无 TTL 无上限，且 amp 重连补发 downSince 之后全部消息，故帧只要到过中继 amp 必收到；而 amp 日志窗口内连续在线且零条记录 → 帧未到中继。根因：手机点 restart 时 wsSend 检查连接非 OPEN 直接返回 false，调用方拿到 false 后静默 return，无任何用户可见提示。上下文用量停在 58% 是其症状，非独立缺陷。agent-meeting 侧对 `control:` 前缀完全透传（全库 16 条 control:restart，最近一条 07-18 发给 atlas，无一条发给 wdav3-laptop）。

**贯穿模式：失败必须响亮。** 本轮独立发现三处同型缺陷——只改出口的静默错投、wsSend 静默吞、注册失败静默瘫痪。共性不是忘加提示，而是**失败路径没有接收者**。

## 1. 目标

amp、daemon、control 三侧一律以 `(name, project)` 复合键标识会话与寻址。

## 2. 分阶段方案（顺序不可颠倒）

> **状态（2026-07-23）**：阶段 -1 / 0 / 2 / 4 已完成；阶段 3 经 Tommy 决定不做。本方案至此收尾。

### 阶段 -1：AMBridge / PWA 侧（amb 负责，是阶段 2 与阶段 0.2 的前置）

- 收件人用 CLI 原生 `name@project` 语法编码进现有 `to` 字符串，不新增字段。
- **入口与出口必须同一 PR**。只改出口是净退步：从响亮失败退化为静默精确错投。
- 手机端 `state.agentsData` 全仓按裸名索引（11 处），收口为 `findAgent(name, project)`；restart 按名回查会重启错人，必修。
- **本地存量键迁移**（与上一条是两回事）：现有 `_migrateProjectKeys()` 用 `name → project` 单值 Map，同名跨项目后项覆盖前项，再合并旧桶、超 300 条截断。反例：本地 `old-a\0sam` 200 条 + `project-b\0sam` 200 条，服务端把前者归一到 `project-a` 后，两段历史被合并并截成 300 条 → 串线 + 丢 100 条。改为按**旧复合键 → 新复合键**逐桶迁移，映射缺失时优雅退化。
- **PWA 版本门禁 + 自我更新**（v3/v4 阻塞项）。现状：`APP_BUILD` 仅用于贴 build 徽章，全仓无其它引用；无任何 fetch 读服务端 `/version`；无 service worker。**后果：一个开着的旧标签页会无限期跑旧 JS，既不自动更新，也无法从服务端观测其版本**。故：
  - 握手帧带 `APP_BUILD`（提供证据）；
  - 自身 build 与服务端 `/version` 不一致时触发重新加载（新客户端的自愈路径）；
  - **协议门禁（关键）**：缺失或不兼容 `APP_BUILD` 的 PWA **不允许发送业务帧**，校验在**每次连接/恢复时**执行，而非一次性枚举。理由：一次性枚举只能证明某时刻的状态，管不住之后从离线/冻结中恢复的旧实例；而旧 JS 既无上报也无自愈，**它不会自己变好**。
  - **门禁能做什么、不能做什么（勿按过于乐观的预期排期）**：真正的旧 JS 读不懂拒绝原因，**不会自动重载**，它只会连接被拒、功能不工作、进入退避重试。门禁的真实价值是 (1) 挡住数据破坏——旧客户端再也发不出裸名帧；(2) 把静默丢数据换成**显性不可用**，用户看得出坏了 → 主动刷新 → 刷新后成为新客户端。**修复路径中含一个消不掉的人工动作（用户刷新）。**
  - 实现须确认：旧客户端被拒后页面上是**可察觉的异常**，而非"页面看起来正常、只是什么都发不出去"——后者等于把静默失败从发送层挪到连接层，白改。
  - 约束：不为旧版本写兼容层；不新增独立轮询定时器（中继单通道连续 6 小时 ≤ 15000 次请求），版本检测搭现有时机的车。
- wsSend 失败必须给用户可见提示，不得静默吞。
- register 返回明确结果、重试有上限、注册被拒可传到用户可见处（现状：register 无返回值、启动注册无限重试无退避上限、失败仅一行日志）。
- 控制转发的 `amp-<host>` 显式补 `@*`（executor 以 `project="*"` 注册）。

### 阶段 0：关闭漂移产生机制 + 存量收敛 + 产出身份重映射表（阻塞后续全部）

**一次性迁移只能清存量，清不掉产生机制。** 现存漂移入口：注册时只有用户显式传才带 `--proj`，缓存按本机仓库根存；新机器 clone 到新路径、缓存为空、用户未传 → 退回路径推导 → 新旧历史互不可见。doc12 §3.2「无 --proj 时用 cwd 兜底」的降级设计本身即该入口。

**0.1 关闭产生机制**（可独立完成，不触碰存量数据）
- **取消静默降级：无法解析权威项目身份时拒绝注册**，提示显式传 `--proj`。不保留路径兜底——响亮失败一次（每机每仓仅首次）优于静默漂移后手工捞数据。二者只能选一。
- 拒绝响应带**机器可读原因码**（至少区分：缺权威项目身份 / 名字被占 / 身份解析失败）。下游按不硬编码白名单方式原样透出。
- **注册时上报 `client_version`**。现状 sessions 表无版本列，注册 payload / roster / 握手帧均无版本，`meeting controls` 的 version 是 daemon 自身的——没有任何手段知道各客户端实际跑什么版本，缺此项则「全机滚完」无法证明。
- 全机（Mac / Windows / 各 codex 会话）滚到含上述改造的版本。
- **本步失败不阻塞**：只要严格停在 0.1、不进入 0.2，部分机器滚版失败继续滚即可，系统状态不变。

**0.2 存量收敛，并产出显式身份重映射表**

v2 的「可达性校验」不可判定：用消息行自身复合键去 read 几乎必然读得回，证明不了手机能从 canonical 入口发现它；用 canonical 键校验则需一张「旧复合键 → canonical 复合键」映射表，而其来源未定义。同理 prune 前置里「应归入 canonical」在数据库层面不是可计算谓词。反例：消息挂 `probeproj@test-project`、canonical 会话是 `probeproj@wda-v3`，用前者读能过验收，手机从后者进去仍看不见。

**修法**：迁移不做「就地改写完即丢」，而是产出并持久化一张显式身份重映射表。

- **必须是函数：每个旧复合键恰好映射到一个新复合键**（v3 阻塞项修正）。v3 的「多值映射」措辞有歧义，被读成允许 `O → {A, B}`——那样挂在 O 上的消息能通过可达性校验却判定不了该迁往哪一个；在 AMBridge 侧更致命：一个旧本地桶对应多个新键时，**那些消息物理上无法拆分**（消息里没有可区分项目的信息）。
- 真出现一对多，必须**下沉到行级判定**，不得在键级留下歧义。
- **必须记录行级来源**（哪些具体行由哪条映射改写），使迁移**可反向**。人工判定必然会错几条，**可反迁移比验收通过更重要**。
- 自动推断覆盖漂移类（同一 canonical 身份的路径派生变体）。
- 不可自动推断的（如 probeproj 这类并非漂移、而是独立脏身份却收过真实消息的），**人工判定一次并写入该表**。人工一次换来永久可判定。
- 该表**可导出**，供 AMBridge 侧本地键迁移消费——两边用同一份真相。格式由本仓定义后知会 amb；在格式确定前 amb 不实现消费逻辑。

有了该表，两个原本说不清的判断变为可查：**孤儿判定** = 端点既不在 canonical 集合中、也未被映射表命中；**可达性校验** = 每条消息两端均能经映射表落到存活 canonical 身份。

**0.3 四表联动迁移 + 中断恢复语义**（v3 阻塞项，本版新增）

四表：messages / sessions / read_cursors / group_members（漏 read_cursors 会 backlog 重放，漏 group_members 会掉群）。

- **优先单事务**：四表改写在同一事务内完成，中断即整体回滚。
- **若无法单事务，则重跑幂等优先于检查点**——中断后最可能的动作就是重跑。迁移须定义为可安全重复执行：已改写的行再次执行不产生二次改写。
- 行级来源记录（见 0.2）同时充当反向记录，使已完成部分可被反迁移。
- 禁止出现「messages/sessions 已改写并发生键合并，而 read_cursors/group_members 未改写」且无法恢复的中间态。

**0.4 与 AMBridge 对时点**：迁移会改变部分行的 project 取值，手机端以 `project+name` 作本地存储键。执行前须确认：amb 的按复合键迁移逻辑已上线，**且 PWA 协议门禁已上线生效**（判据是门禁生效，不是「旧客户端已清零」——后者永远等不到，存量旧标签页只会以用户发现异常后刷新的速度自然消退，时间不可控）。

**验收门槛（6 条，缺一不可）**
1. 全库无旧路径派生的 project 值残留。
2. 任意 monitor 重启 / 重连不产生新漂移行；未提供权威身份时注册被**拒绝**而非降级，且拒绝原因码可读。
3. 四表外键校验干净。
4. **可达性校验（经重映射表判定）**：全库每条 message 两端均能经映射表落到存活 canonical 身份。前 3 条验的是**引用完整性**，验不出可达性——messages 无指向 sessions 的外键，孤儿必然通过前 3 条。
5. 客户端版本上报已生效，可枚举各 agent 实际版本；**PWA 协议门禁已上线生效**（不合格客户端无法发送业务帧）。
6. 重映射表为**函数**（无一对多键级歧义），且行级来源记录完整、可支撑反迁移。

### 阶段 3：硬门禁（防复发）—— **不做（Tommy 决定，2026-07-23）**

> **本阶段已由 Tommy 拍板不做。** 下面的分析与教训保留作记录，不再作为待办。已知代价：测试隔离靠自觉遵守而非强制——新写的测试需自行同时设 `MEETING_HOME` 与 `MEETING_HOST`，并避免「未识别子命令转发到真实 CLI」的桩。`agent-meeting/tests/test_prune.py` 是遵守该约定的参考样例。

- 测试入口检测未设 MEETING_HOME 重定向即拒跑。probeproj / zz-restart-test 均由未隔离的测试直接打在正式库上产生，且项目身份缓存以仓库根为键会「粘住」，清了会再生。

**2026-07-22 补充：上面这条不够，已实测证伪。** `MEETING_HOME` 只重定向本地文件路径（db、config.json、pidfile），挡不住网络。`meeting` CLI 的 `online`/`offline`/`controls` 等子命令解析目标 daemon 时走独立的 `_resolve_host()`：env `MEETING_HOST` → 用户钉住的 `config.json control_host` → **mDNS/LAN 实时发现** → TCP 探活的自动缓存。测试若只设了 `MEETING_HOME`（哪怕是全新的空目录，没有 `config.json`），前两级都落空，会直接掉进 mDNS 发现——这是对局域网的真实广播查询，跟 `MEETING_HOME` 完全无关，会把测试子进程带到同一台机器/局域网上正在跑的真实 daemon 上注册。`test_ws_monitor.py` 一次事故：测试自带的假 `meeting` CLI 只接管了 `controls --json`（供 WS 连接发现用），其余子命令（含 `online`/`offline`）直接 `execv` 转发到 `~/.agent-meeting/bin/meeting`（机器上装的真实版本），该真实版本收到转发调用时未带 `--host`，于是走 mDNS 发现命中生产 daemon，把 8 个测试身份注册进了生产库——测试全绿（因为 WS 收发路径被正确指向了测试 daemon），污染照样发生。

修复后的门禁应为**两条**，缺一不可：
1. 文件路径隔离：`MEETING_HOME` 重定向到临时目录（原有要求，仍然必要）。
2. **网络路径隔离**：任何会跑 `meeting` CLI 子命令（含 monitor.py/codex-bridge.py 内部转调）的测试子进程，必须在同一份 env 里同时设 `MEETING_HOST` 指向测试自己起的 daemon。`MEETING_HOST` 在 `_resolve_host()` 精度链里排第一位，命中后直接短路，不会触达 mDNS 分支——这是唯一对所有子命令都生效、且不需要逐个子命令传 `--host` 的钉法。「测试入口检测未设 MEETING_HOME 重定向即拒跑」要扩展为「检测未设 MEETING_HOME **或** MEETING_HOST 即拒跑」（响亮失败，不允许静默降级到发现路径）。
3. 附带教训：测试里为了打桩而写「未识别子命令一律转发到真实 CLI」的 fallback shim 天然危险——转发目标一旦是机器上装着的真实二进制，就绕开了测试对自身逻辑的隔离。更稳的做法是把源码树里的真实 `bin/meeting`（+`meeting_common.py`)复制一份到隔离目录里跑，而不是转发到 `~/.agent-meeting`。

### 阶段 4（不在关键路径）：清理脏数据 —— **已完成（2026-07-23）**

**已移出关键路径**：清脏数据不是收窄寻址的安全前置——阶段 2 的安全前置是身份口径已归一，脏行留着只是难看，不影响收窄后的正确性。且等门禁与收窄都稳定后再清，脏行边界更清楚，比现在清更不容易误伤。

- `meeting prune` 已实现：daemon 侧 `_prune()` + `POST /prune`，CLI 侧 `meeting prune [--older-than N] [--include-referenced] [--yes]`。默认 dry run，加 `--yes` 才真删。只删 sessions 行，不动 messages。

- **前置检查改了口径（推翻原方案）**：原方案写的是「被清身份若在 messages 中仍被引用、且未被重映射表覆盖 → 阻断，要求先补映射」。实测证伪：库里 `plugins-win@plugins` 有 114 条消息、零重映射记录，按原规则会被拦下并要求补映射，但它只是一台离线的真实会话，没有任何映射可补——原规则会让 prune 在正常离线会话上永久卡死，且「补映射」这个补救动作本身无意义。
  改为两条结构性守门：
  (a) **在线的永不入选**：cutoff 被钳到至少 `ONLINE_THRESHOLD` 之前，`--older-than=0` 也删不掉正在心跳的会话。
  (b) **消息里出现过的默认跳过**，除非显式 `--include-referenced`（且即便强清也只删 sessions 行，messages 一条不动）。
  理由：sessions 行是唯一记录该身份「住在哪台机 / 哪个目录 / 什么角色」的地方，messages 只有复合键没有这些出处信息；删注册行留消息 = 丢出处换不到任何东西。而真正的测试残渣（注册完一条消息都没发过的）消息数为 0，照样自由清除。

## 3. 风险

- 阶段顺序颠倒会立即隐藏分裂的历史会话。收窄寻址只改查询不改数据，事后若映射明确仍可恢复，不构成不可逆损坏——但恢复需人工确定映射，成本高。
- 阶段 2 改动面覆盖读 / 删 / 轮次 / 翻页共用寻址逻辑，回归面广。
- 阶段 0.2 中人工判定部分的正确性无法被自动校验，是本方案唯一依赖人的环节；缓解手段是行级来源记录使其**可反向**，而非试图保证判定不出错。
- 取消路径兜底属有意 breaking change（内部工具，不留兼容层）。
- 存量旧 PWA 标签页的消退速度不可控，依赖用户主动刷新。门禁保证它们不再破坏数据，但不保证它们消失；任何以「旧客户端已清零」为条件的排期都会无限期等待。

## 4. 本轮范围外（已记录）

- 中继 down 队列无 TTL、无上限，随时间线性增长。

## 5. 本轮方法论记录

本轮所有有效评审意见几乎都属同一类手法：**把「某时刻为真」的断言，换成「每次都校验」的持续检查**。两个代表例——
- 把 prune 前置里「应归入 canonical」这个数据库算不出来的谓词，转成「查显式重映射表是否命中」；
- 把「验收时活跃客户端均为兼容版本」这个一次性枚举，转成「每次连接都校验 APP_BUILD」的协议门禁。

另一条贯穿模式是**失败必须响亮**：本轮独立发现四处同型缺陷——只改出口的静默错投、wsSend 静默吞、注册失败静默瘫痪、门禁若做成「连不上但页面正常」。共性不是忘加提示，而是**失败路径没有接收者**。
