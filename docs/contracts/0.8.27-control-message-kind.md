# 0.8.27 — 控制消息结构化 kind（消灭陈旧重放 + 指令/提及歧义）

日期：2026-06-23 PDT
作者：plugins-wic（director）
对接：amb-wic（amp / PWA 发送侧）

## 背景 / 根因

`[restart]` / `[clear]` 这类"只在当下有效"的编排指令，过去靠**内联魔法文本**约定——发送方在消息正文里打 `[restart]`，接收方的 LLM 读到后自行决定跑 `/handoff`。全库无任何代码解析它们。两个结构性缺陷：

1. **陈旧重放**：meeting 是持久补投队列（离线上线补收，对普通消息正确）。但编排指令也被补投——31h 前发给离线 plugins-wic 的 `[restart]`（msg 360）在其重启上线时被重放、错误执行了一次 `/handoff`，当下并无重启。
2. **指令 vs 提及无法区分**：内联文本没有类型，原理上分不清"这是要我执行的指令"还是"这条消息在引用/讨论这个命令"。例：一条 bug 报告正文里字面写了 `[restart]`，前缀/包含匹配会误触发。

## 决策

把控制指令从内联文本**升格为结构化信号**，复用现成的 `--kind` 字段。接收方**只认 kind、不认正文**，并叠一道新鲜度闸。**daemon 保持哑传输，不解析语义、不做时间过滤**（避免层级泄漏、避免误杀合法离线普通消息）。

## 契约

### ① kind 取值（发送侧）

控制指令用以下 kind 发送，正文为人类可读说明（**正文不参与触发判定**）：

- `--kind=control:restart` — 请求接收方跑 `/handoff` 写交接卡后被重启
- `--kind=control:clear` — 请求接收方清场/结束当前会话

格式：`control:<action>`，全小写，冒号分隔。未来新动作沿用 `control:<action>` 命名空间。

### ② 接收侧（meeting skill）触发判定

一条 incoming 消息被当作控制指令执行，**当且仅当**同时满足：

1. **结构化**：`kind` ∈ `control:*` 命名空间（从 `meeting read/show` 读 kind，**绝不从正文文本判断**）。正文里出现 `[restart]`/`[clear]` 等字样一律视为普通文本、不触发。
2. **新鲜**：`now - created_at ≤ 600 秒`（10 分钟硬上限）。**且**不早于本会话 monitor 上线时间（该指令发出时本会话实例还不存在 → 不是发给我的）。
   - 机械判定：`date +%s` 减消息 `created_at`（epoch）。> 600 → 陈旧，忽略。
   - 陈旧时：不执行，向用户输出一行"忽略陈旧控制指令（<动作>，<N>分钟前）"，跳过。

新鲜的 `control:restart` → 跑 `/handoff` 后停止接新活；新鲜的 `control:clear` → 按 clear 语义处理。

### ③ daemon / CLI

- **无需改动**：`--kind` 已是自由文本、原样存储与投递。`control:*` 直接走现有通道。
- daemon 不做新鲜度过滤（陈旧 control 仍会被补投，由接收侧 skill 闸掉）。

### ④ 版本

agent-meeting `0.8.26 → 0.8.27`（仅 meeting SKILL.md 改动 + plugin.json bump）。

### ⑤ 发送侧落地（amp / PWA，amb-wic）

Tommy 经 amp/PWA 触发重启/清场时，改用 `--kind=control:restart` / `control:clear` 发送，**不再内联 `[restart]`/`[clear]` 文本**。无需在正文塞时间戳——`created_at` 是 daemon 服务端字段，已有。

## 边界

- 仅约束编排控制指令；普通消息（回应/知会/总结/询问/报bug 等）行为不变，离线补收照常。
- 历史消息不回改（append-only）；旧的内联 `[restart]` 文本从 0.8.27 起一律按普通文本对待、不再触发。
